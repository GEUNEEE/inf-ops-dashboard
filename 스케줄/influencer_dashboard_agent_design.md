# 인플루언서 대시보드 자동화 에이전트 설계서

> Claude Code 구현 참조용 계획서  
> 작성 기준일: 2026-05-14  
> 최종 수정일: 2026-05-15 (인터뷰 반영 — 클라우드 조회·구간 정산·시계열·카카오톡 알림 도입)  
> 확장 방향: **로컬 자동화 + 정적 웹 대시보드 + 카카오톡 알림**

## 1. 작업 컨텍스트 문서

### 1-1. 배경 및 목적

유튜브 인플루언서 협찬·광고 운영 과정에서 발생하는 네 가지 업무 흐름을 단일 시스템으로 통합한다.

- **메일 발송 현황**: 1,200건 이상의 발송 내역과 퍼널 전환율을 자동 집계
- **인플루언서 관리**: 체험·광고 단계별 진행 현황을 인플루언서관리 시트에서 직독
- **주문·정산**: 스마트스토어 주문 파일을 업로드하면 DB에 반영하고 인플루언서별 정산서를 자동 발행
- **대시보드 조회**: 외부에서 모바일/PC로 매출·진행상태·KPI 추세를 언제 어디서든 조회

### 1-2. 범위

|------|------|

### 1-4. 제약조건

- 스마트스토어 주문 파일은 `CDFV2 암호화` 상태 → 비밀번호 입력 후 복호화 필수. 모든 파일 비밀번호 동일 → `.env`에 1회 저장
- 주문 파일의 `유튜버 이름` 컬럼은 수식으로 작성 → `data_only=True`로 읽으면 수식 문자열 반환 → **상품명에서 정규식으로 유튜버명 추출**
- `우리의 서술집`은 완전 제외 대상 — Raw_Data 반영 자체를 차단
- 기타(검토·진행불가·우리측거절) 상태는 퍼널 분모·분자 모두 제외, 별도 표기
- **인플루언서관리 시트에 미등재된 인플루언서는 정산 계산에서 제외** — Raw_Data 반영은 하되 정산서 미발행, 대시보드 주문 목록에는 표시
- **인플루언서 미연결 일반 판매 건(기타/일반)은 정산 합계에 별도 항목으로 포함** — 개별 정산서는 발행하지 않음
- **정산 단가는 누적 수량 구간**으로 적용 (Raw_Data 전체 기준, 월 리셋 없음)
- **환불·반품 발생 시 익월 정산에서 차감** — 당월 발급된 정산서는 확정값 보존
- **부가세는 표기만** — 정산서에 공급가액 그대로 + "VAT 10% 제외" 칼럼 병기 (세금계산서 발행 별도)
- **정산서 발송은 사용자가 시트별 발췌** — 자동 발송 아님. 인플루언서별 시트는 반드시 분리 (타 인플루언서 정보 포함 금지)
- 웹 대시보드는 **조회 전용** — 업로드·설정 변경은 로컬 PC에서만
- git push 실패 시 1회 재시도 → 실패 시 로컬 로그만 남기고 계속 진행

### 1-5. 용어 정의

| 용어 | 정의 |
|------|------|
| **퍼널** | 총발송 → 응답 → 미팅 → 체험 → 광고 단계별 전환 흐름 |
| **응답** | 회신일 컬럼에 날짜가 있거나 진행상태가 기재된 건 |
| **체험전환** | 인플루언서관리 시트에 등재된 인원 (협찬수락일 기준) |
| **기타** | 검토·진행불가·우리측거절 — 퍼널 계산에서 완전 제외 |
| **Raw_Data** | 정산 파일의 주문 원본 시트 (중복 방지 기준) |
| **협찬수락일** | 인플루언서가 체험에 동의한 날짜 (체험진행 카운팅 기준) |
| **managed_set** | 인플루언서관리 시트 행11(유튜버명)에서 추출한 등재 인플루언서 이름 집합 |
| **기타/일반** | 상품명에 `[OOO 구독자 대상]` 패턴이 없는 주문 — 인플루언서 미연결 일반 판매 건 |
| **관리탭 미등재 인플루언서** | 주문 파일에 유튜버명이 있으나 managed_set에 없는 경우 — Raw_Data 반영 O, 정산서 미발행 |
| **완전 제외** | `exclude` 목록에 있는 인플루언서 — Raw_Data 반영 자체를 차단 |
| **누적 수량** | 정산 시점까지 해당 인플루언서가 Raw_Data에 누적된 수량 (월 리셋 없음) |
| **구간 단가** | 누적 수량에 따른 개당 정산 단가 — 1–29개 ₩20,000 / 30–99개 ₩22,000 / 100개+ ₩25,000 |
| **개인 정산** | 운영자(본인)가 주문당 수령하는 ₩10,000 인건비/수수료 |
| **기타 비용** | 제품원가·택배·스마트스토어 수수료·광고비 합산 — 현재 주문당 ₩50,000 고정 |
| **매출** | 해당 기간 스마트스토어 주문금액 합계 |
| **수익** | 매출 − (인플루언서 정산 + 개인 정산 + 기타 비용) |
| **월별 스냅샷** | 주문 처리 완료 시 `site/data/history/YYYY-MM.json` 에 저장되는 월 집계 데이터 — 추세선 소스 |

## 2. 워크플로우 정의

### 2-1. 전체 흐름도

```
[로컬 PC — watchdog 자동 감지]
      │
      ├─ /input 폴더에 파일 드롭 감지
      │
      ├─ 마스터 DB (인플루언서관리.xlsx) 감지
      │       │
      │       ├─→ [STEP 1] 메일발송현황 파싱 → 퍼널 KPI 계산
      │       └─→ [STEP 2] 인플루언서관리 시트 파싱 → managed_set 추출
      │
      └─ 주문 파일 (스마트스토어.xlsx) 감지
              │
              ├─→ [STEP 3] 비밀번호 복호화 (.env에서 자동 로드)
              ├─→ [STEP 4] 중복 체크 & Raw_Data 반영 & 버킷 분류
              ├─→ [STEP 5] 정산서 생성 (구간 단가 + VAT 칼럼)
              ├─→ [STEP 6] 매출·수익 집계
              ├─→ [STEP 7] 월별 스냅샷 저장 (history/YYYY-MM.json)
              ├─→ [STEP 8] 카카오톡 메모챗 알림 (MCP)
              └─→ [STEP 9] site/data/*.json 갱신 → git push → GitHub Pages 반영

[STEP 0 — 과거 주문 일괄 임포트 (1회성)]
  2026-03 이후 보유 주문 파일 시간순 처리
    → Raw_Data 재구축 → 월별 스냅샷 재구성 → site/data 전체 갱신

[정적 사이트 — GitHub Pages]
  index.html → KPI 카드 + 인플루언서 그리드 + 메일 추세
  influencer.html?name=... → 드릴다운 (월별 추이·누적·향후 일정)
```

### 2-2. 단계별 상세

#### STEP 0 — 과거 주문 일괄 임포트 (1회성)

| 항목 | 내용 |
|------|------|
| **처리 방식** | 수동 트리거 (Claude Code에 "과거 임포트 실행" 요청 시) |
| **입력** | 2026-03 이후 보유한 스마트스토어 주문 xlsx 파일 목록 |
| **비밀번호** | `.env`에서 자동 로드 (전 파일 동일) |
| **순서** | 파일명 날짜 순 오름차순 처리 — 중복 스킵 로직 동일 적용 |
| **출력** | Raw_Data 재구축 + `history/2026-03.json`, `history/2026-04.json` 등 생성 |
| **성공 기준** | 처리 파일 수 = 입력 파일 수, Raw_Data 총 행 증가 확인 |

#### STEP 1 — 메일발송현황 파싱 및 퍼널 계산

| 항목 | 내용 |
|------|------|
| **처리 방식** | 스크립트 (openpyxl, data_only=True) |
| **입력** | 메일발송현황 시트 (행7~끝, 열25=발송일, 열27=회신일, 열28=진행상태, 열30=협찬수락일) |
| **로직** | 발송일 있는 행만 대상 / 진행상태=기타인 행은 분모·분자 제외 |
| **응답 판단** | 회신일 OR 진행상태 기재 OR 협찬수락일 있음 |
| **미팅 단계** | 미팅대기 / 미팅진행 2단계 (미팅예정은 미팅대기로 통합) |
| **성공 기준** | 5개 KPI 값 모두 0 이상의 숫자로 산출 |
| **검증 방법** | 규칙 기반 — 응답률 ≤ 100%, 각 단계 건수 합계 ≤ 총발송 |
| **실패 처리** | 자동 재시도 1회 → 에스컬레이션 |

**KPI 계산 공식**

```
총발송        = 발송일 있는 행 중 진행상태 ≠ 기타인 건수
응답률        = (회신일 있음 OR 진행상태 기재) / 총발송
미팅전환율    = (미팅대기 + 미팅진행) / 총발송
체험전환율    = 인플루언서관리 시트 등재 인원 / 총발송   ← STEP 2 기준
광고수락률    = 광고단계(예정+완료) 인원 / 총발송        ← STEP 2 기준
```

#### STEP 2 — 인플루언서관리 시트 파싱

| 항목 | 내용 |
|------|------|
| **처리 방식** | 스크립트 (openpyxl) |
| **구조** | 전치형 (행=항목, 열=인플루언서) — 행11=유튜버명, 행10=현재상태, 행24=체험수락일 등 |
| **진행상태 분류** | 1차 체험진행 / 2차 체험진행 / 3차 체험진행 / 1차 광고예정 / 2차 광고예정 / 1차 광고완료 / 기타 |
| **핵심 산출물** | `managed_set` — 관리탭 등재 유튜버명 집합, STEP 4·5의 정산 대상 판별에 사용 |
| **성공 기준** | 유튜버명 1개 이상 파싱, 현재상태 필드 존재 |
| **검증 방법** | 스키마 검증 — managed_set 원소 수 ≥ 1 |
| **실패 처리** | 에스컬레이션 (시트 구조 변경 가능성) |

#### STEP 3 — 주문 파일 복호화

| 항목 | 내용 |
|------|------|
| **처리 방식** | 스크립트 (msoffcrypto-tool) |
| **입력** | 암호화 xlsx + `.env`에서 자동 로드한 비밀번호 |
| **출력** | 복호화된 임시 파일 (`/input/order_decrypted.xlsx`) |
| **성공 기준** | openpyxl로 정상 로드 가능 |
| **검증 방법** | 규칙 기반 — 시트명 `주문조회` 존재 확인 |
| **실패 처리** | 에스컬레이션 (비밀번호 오류 메시지 출력) |

#### STEP 4 — 중복 체크 및 Raw_Data 반영 + 버킷 분류

| 항목 | 내용 |
|------|------|
| **처리 방식** | 스크립트 (openpyxl) |
| **중복 기준** | 상품주문번호 (열0) 기준 기존 Raw_Data와 대조 |
| **제외 조건** | 주문상태=취소 OR 클레임상태=취소완료 |
| **유튜버명 추출** | 상품명에서 정규식 `\[([^\]]+)\s+구독자` 패턴으로 추출 후 name_map 정규화 |
| **버킷 분류 기준** | 아래 결정 트리 참조 |
| **성공 기준** | 신규 건수 ≥ 0 (0건도 정상 — 이미 반영된 경우) |
| **검증 방법** | 규칙 기반 — 추가 후 Raw_Data 총 행수 = 기존 + 신규 건수 |
| **실패 처리** | 스킵 + 로그 (개별 행 파싱 오류 시) |

**주문 버킷 분류 결정 트리**

```
주문 1건 입력
    │
    ├─ 취소 여부 확인
    │   ├─ 취소 → [스킵] Raw_Data 미반영
    │   └─ 유효 → 다음
    │
    ├─ 유튜버명 추출 (상품명 정규식)
    │   ├─ 패턴 없음 → [기타/일반] Raw_Data 반영 O | 정산 별도 합산 항목
    │   └─ 패턴 있음 → name_map 정규화 → 다음
    │
    ├─ exclude 목록 확인
    │   ├─ 해당 (우리의 서술집 등) → [완전 제외] Raw_Data 미반영 | 스킵
    │   └─ 해당 없음 → 다음
    │
    └─ managed_set 등재 여부 확인
        ├─ 등재 O → [정산 대상] Raw_Data 반영 O | 개별 정산 시트 발행
        └─ 등재 X → [정산 제외] Raw_Data 반영 O | 정산서 미발행 | 로그 기록
                                                   + 카카오톡 즉시 푸시
```

**유튜버명 정규화 매핑 (name_map)**

```python
YTBER_MAP = {
    '마성민 러닝라이프': '마성민',
    '마성민 라이프러너': '마성민',
    'The 조치 패밀리':   'The 조치 패밀리',
}
EXCLUDE = {'우리의 서술집'}
```

#### STEP 5 — 정산서 생성 (구간 단가 + VAT)

| 항목 | 내용 |
|------|------|
| **처리 방식** | 스크립트 (openpyxl + 서식 적용) |
| **입력** | STEP 4 버킷 분류 결과, `ytber_config.json`의 `tier_pricing`, managed_set |
| **단가 계산** | Raw_Data 전체에서 인플루언서별 누적 수량 집계 → 구간 단가 적용 |
| **구간 단가** | 1–29개 ₩20,000 / 30–99개 ₩22,000 / 100개+ ₩25,000 (누적 기준, 경계 주문부터 새 단가) |
| **[정산 대상] 처리** | `{유튜버명} {월}월` 시트 생성 — 독립 시트, 타 인플루언서 정보 미포함 |
| **시트 구성** | ① 기본정보 (인플루언서명·정산월·누적수량·현재단가) ② 주문 상세 테이블 ③ 공급가 합계 ④ VAT 10% 제외 금액 (공급가 ÷ 1.1, 소수점 절사) |
| **환불 처리** | 당월 정산서는 확정값 — 환불은 익월 정산서에 차감 행으로 별도 표시 |
| **[기타/일반] 처리** | `기타일반 {월}월` 시트 — 동일 양식, 파트너명란에 `기타/일반`, 금액란 `null` |
| **[정산 제외] 처리** | 시트 미생성 + `settlement_skipped.log` 기록 |
| **성공 기준** | 정산 대상 N개 + 기타일반 1개 시트 생성 완료, 합계 금액 = 수량 × 단가 |
| **검증 방법** | 규칙 기반 — 시트 수 확인, 합계 금액 재계산 대조, VAT 칼럼 = 합계 ÷ 1.1 |
| **실패 처리** | 스킵 + 로그 (개별 인플루언서 오류 시 해당 시트만 건너뜀) |

#### STEP 6 — 매출·수익 집계

| 항목 | 내용 |
|------|------|
| **처리 방식** | 스크립트 |
| **입력** | 이번 달 Raw_Data 신규 행, `ytber_config.json` 비용 설정 |
| **매출** | 신규 주문금액 합계 (취소·기타/일반 포함, 완전제외 제외) |
| **인플루언서 정산 비용** | 구간 단가 × 수량 (STEP 5에서 계산된 값 재사용) |
| **개인 정산 비용** | `personal_fee_per_order` × 주문 건수 |
| **기타 비용** | `fixed_misc_cost_per_order` × 주문 건수 |
| **수익** | 매출 − (인플루언서 정산 + 개인 정산 + 기타 비용) |
| **출력** | `revenue` 딕셔너리 — dashboard.json에 삽입 |

```
수익 = 매출 - Σ(구간 단가 × 인플루언서별 수량) - (10,000 × 주문수) - (50,000 × 주문수)
```

#### STEP 7 — 월별 스냅샷 저장

| 항목 | 내용 |
|------|------|
| **처리 방식** | 스크립트 |
| **트리거** | 주문 처리 완료 시마다 해당 월 스냅샷 갱신 |
| **출력 경로** | `site/data/history/YYYY-MM.json` |
| **포함 데이터** | 월 매출·수익·주문수·수량·KPI 5종·인플루언서별 집계 |
| **정책** | 같은 달 재처리 시 덮어쓰기 (최신값으로 갱신) |

#### STEP 8 — 카카오톡 메모챗 알림

| 항목 | 내용 |
|------|------|
| **처리 방식** | MCP `KakaotalkChat-MemoChat` 호출 |
| **트리거** | (A) 주문 처리 완료 (B) 관리탭 미등재 인플루언서 발견 즉시 |
| **메시지 구성** | ① 주문 처리 요약 (신규 N건, 정산 M명, 기타 K건) ② 관리탭 미등재 목록 (있을 때만) ③ 이번 달 매출·수익 쇼트 (₩XXX / 수익 ₩YYY) ④ 대시보드 URL |
| **미등재 알림** | 발견 즉시 별도 메시지 선발송 — 이름·건수·수량 포함 |
| **실패 처리** | 1회 재시도 → 실패 시 로컬 로그에만 기록 후 계속 |

#### STEP 9 — 정적 사이트 동기화 (git push)

| 항목 | 내용 |
|------|------|
| **처리 방식** | 스크립트 (git CLI) |
| **순서** | `site/data/` 갱신 → `git add site/data/` → `git commit -m "data: YYYY-MM-DD 주문 반영"` → `git push` |
| **실패 처리** | 1회 재시도 → 실패 시 로컬 로그 남기고 계속 (다음 주문 처리 시 재시도) |
| **캐시 무효화** | `dashboard.json?v={timestamp}` 쿼리스트링으로 클라이언트 캐시 무효화 |

### 2-3. 분기 조건 및 상태 전이

```
주문 파일 업로드 여부
    ├─ YES → STEP 3~9 실행
    └─ NO  → 마스터 DB 변경 감지 시 STEP 1~2 + STEP 9 (KPI만 갱신)

복호화 성공 여부
    ├─ YES → STEP 4 진행
    └─ NO  → 에스컬레이션 (비밀번호 오류 메시지 출력, .env 확인 요청)

신규 주문 건수
    ├─ 0건  → "이미 반영된 파일입니다" 메시지 출력 후 종료
    └─ 1건+ → STEP 5~9 진행

주문 버킷 분류 결과
    ├─ 정산 대상       → 개별 정산 시트 발행 (구간 단가 적용)
    ├─ 기타/일반       → 기타일반 시트에 합산 (금액 null)
    ├─ 정산 제외       → Raw_Data 반영 + 정산서 미발행 + 로그 + 카카오톡 즉시 알림
    └─ 완전 제외       → Raw_Data 미반영 + 스킵

구간 단가 경계 통과 (예: 29→30번째 주문)
    └─ 그 주문부터 새 단가 적용 (부분 분할 없음)

환불·반품 발생
    └─ 당월 정산서 수정 없음 → 익월 정산서에 차감 행으로 별도 표시

git push 실패
    ├─ 1회 재시도
    └─ 실패 시 로컬에만 저장 (다음 처리 시 재시도)

인플루언서 정산 단가 미등록 (ytber_config 없음)
    ├─ default_unit_price(₩20,000) 적용
    └─ 경고 메시지 출력 후 계속 진행
```

### 2-4. LLM 판단 vs 코드 처리

| LLM 판단 영역 | 코드 처리 영역 |
|--------------|--------------|
| 파일 구조 변경 감지 시 파싱 전략 결정 | xlsx 파싱 및 집계 |
| 진행상태 신규 값 등장 시 분류 판단 | 중복 체크 로직 |
| 카카오톡 알림 메시지 톤 조정 | 구간 단가 계산 (tier_pricing 적용) |
| 유튜버명 매핑 불일치 시 후보 제안 | 매출·수익 산식 |
| 매출 이상치 발견 시 코멘트 생성 | 스냅샷·git 동기화 |
| managed_set 미등재 인플루언서 발견 시 카카오톡 메시지 작성 | 암호화 파일 복호화 |
| 구간 정책 변경 감지 (config 불일치) | 주문 버킷 분류 로직 |

### 3-2. CLAUDE.md 핵심 섹션 목록

- **수동 모드 (과거 임포트)**: 파일 목록 수동 지정, STEP 0 로직 순차 실행

|------|------|
| **입력** | 암호화 xlsx 경로, `.env` 비밀번호, 마스터 DB 경로, `managed_set` (JSON 직렬화 인라인 전달) |
| **출력** | `/output/정산DB_업데이트.xlsx`, `/output/settlement_skipped.log`, `revenue` 딕셔너리 |
| **참조 스킬** | `excel-parser`, `settlement-generator`, `dashboard-builder(build_revenue)` |
| **데이터 전달** | 파일 기반 — 복호화 결과를 `/input/order_decrypted.xlsx`에 저장 후 경로 전달 |
| **성공 조건** | 정산 대상 N개 + 기타일반 1개 시트 생성 완료, revenue 딕셔너리 반환 |
| **실패 조건** | 비밀번호 오류, 시트 구조 미매칭 → 메인 에이전트에 에스컬레이션 |

### 3-6. 주요 설정 파일: ytber_config.json

```json
{
  "name_map": {
    "마성민 러닝라이프": "마성민",
    "마성민 라이프러너": "마성민"
  },
  "exclude": ["우리의 서술집"],
  "tier_pricing": [
    { "min_qty": 1,   "unit_price": 20000 },
    { "min_qty": 30,  "unit_price": 22000 },
    { "min_qty": 100, "unit_price": 25000 }
  ],
  "personal_fee_per_order": 10000,
  "fixed_misc_cost_per_order": 50000,
  "vat_rate": 0.10,
  "general_sales_label": "기타/일반"
}
```

> `tier_pricing`은 `min_qty` 오름차순으로 정렬. 누적 수량이 해당 구간 이상이면 그 단가 적용.  
> `exclude` 목록은 Raw_Data 반영 자체를 차단한다.  
> 관리탭 미등재 여부는 런타임에 `managed_set`과 대조하여 판별하며, 이 파일에서 관리하지 않는다.

## 5. 클라우드 조회 아키텍처

### 5-1. 토폴로지

```
[로컬 PC]
  watchdog → 파이프라인 실행
  → site/data/*.json 갱신
  → git push
      │
      ▼
[GitHub 비공개 리포지토리]
  (추측 불가 이름 권장)
      │
      ▼ GitHub Pages
[정적 사이트 URL]
  (robots.txt + meta noindex로 검색엔진 색인 차단)
      │
      ├──────────────────────────────┐
      ▼                              ▼
[PC 브라우저]               [모바일 브라우저]
index.html (풀 대시보드)    index.html (반응형 카드 레이아웃)
influencer.html (드릴다운)  influencer.html (스크롤 기반)
```

### 5-2. 보안 모델

| 수준 | 방법 | 상태 |
|------|------|------|
| 기본 | 추측 불가 리포지토리 이름 + 비공개 GitHub Pages URL | 현재 적용 |
| 기본 | `robots.txt` Disallow: / + `<meta name="robots" content="noindex">` | 현재 적용 |
| 선택 | Vercel + Google OAuth (특정 이메일 화이트리스트) | 필요 시 전환 |

> 현재 보안 수준은 "링크를 아는 사람만 접근"이므로, 정산 데이터 민감도가 높아지면 OAuth 전환을 권장한다.

### 5-6. 다크모드 및 UI 스택

- **다크모드**: Tailwind `dark:` 토큰, 시스템 설정 자동 감지 (`prefers-color-scheme: dark`)
- **차트**: Chart.js (MIT) — 추세선은 `line`, 매출/수익은 `bar`+`line` 혼합
- **톤**: 다크 배경(`gray-900`) + 포인트 컬러(`indigo-400`) + 수치 강조(`white bold`)
- **빌드 도구**: 없음 (CDN 방식) — 정적 사이트 배포 단순화
